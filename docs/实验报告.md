# 灵感笔记 HarmonyOS 应用实验报告

## 一、项目概述

### 1.1 项目基本信息

- 项目名称：灵感笔记（todolistAPP）
- 应用包名：`com.example.todolistapp`
- 开发平台：HarmonyOS
- SDK 版本：6.0.2（API 22）
- 开发语言：ArkTS、C++
- 构建工具：Hvigor、CMake、Ninja
- 工程模式：Stage Model
- 目标设备：手机、平板

灵感笔记是一款面向个人用户的轻量级笔记与待办管理应用。应用核心目标是降低用户记录和整理信息的成本：用户输入会议记录、学习笔记、生活备忘或临时想法后，系统可以自动提取关键词、推荐标签、生成摘要，并在文本中存在明确行动项时生成待办事项。

项目采用“本地优先、云端辅助”的 AI 整理策略。本地 AI 由 ArkTS 规则逻辑与 C++ Native 模块共同实现，可在离线场景下完成基础分析；云端 AI 通过通义千问兼容接口补充更强的生成能力。实验重点围绕 Native 原生增强和全场景适配展开，完成了 C++ 文本分析、C++ 密码加密、响应式布局和平板分栏四个方向。

### 1.2 实验目标

本实验主要完成以下目标：

1. 使用 HarmonyOS ArkTS 实现笔记、待办、搜索、图片、用户登录注册等基础功能。
2. 将核心文本分析能力迁移到 C++，通过 NAPI 暴露给 ArkTS 调用，改善中文关键词提取效果。
3. 将用户密码存储由明文改为 C++ SHA-256 加盐哈希，提高本地数据安全性。
4. 面向手机和平板实现响应式布局，避免简单等比拉伸。
5. 在平板设备上实现列表-详情双栏结构，提高大屏使用效率。
6. 完成集成测试、HapTest 测试方案整理和实验报告编写。

### 3.2 数据模型设计

项目主要使用 `notes` 和 `todos` 两张表。

`notes` 表保存笔记和待办的通用字段，包括标题、正文、图片、标签、AI 摘要、行动项、优先级、创建时间、是否待办和待办状态等。普通笔记与待办通过 `isTodo` 字段区分。

`todos` 表保存待办详情字段，包括关联笔记 ID、状态、截止时间和优先级。该设计方便后续扩展提醒、截止日期和更多待办属性。

`Note` 模型中核心字段包括：

```typescript
interface Note {
  id: string
  userId?: string
  title: string
  content: string
  imagePath: string
  tags: string
  aiTags?: string
  summary?: string
  actionItems?: string
  priority?: number
  createTime: number
  isTodo: number
  todoStatus: TodoVisualStatus
}
```

### 3.3 本地 AI 数据流

本地 AI 整理流程如下：

1. 用户在 `NoteEditPage` 输入标题和正文。
2. 点击“本地 AI 自动提取”。
3. 页面调用 `NoteService.generateLocalAiSuggestion(title, content)`。
4. `NoteService` 调用 Native 文本分析模块提取关键词和摘要候选句。
5. ArkTS 规则层继续完成标签推荐、摘要压缩、行动项抽取、时间识别和优先级推断。
6. 页面展示关键词、推荐标签、摘要和待办行动项。
7. 如果存在明确行动项，用户可选择保存为待办；如果是纯记录，则不强制生成待办。

## 四、功能实现

### 4.1 用户系统

用户系统由 `LoginPage`、`RegisterPage` 和 `UserViewModel` 实现。用户注册时输入账号、昵称和密码，系统调用 Native 加密模块生成随机盐值并计算 SHA-256 哈希，最终在 Preferences 中保存 `hash:salt` 格式记录，而不是明文密码。

登录时，`UserViewModel` 从 Preferences 读取用户记录。如果记录已经是 `hash:salt`，则调用 `NativeCrypto.verifyPassword()` 完成校验；如果检测到旧版明文记录，则在密码正确时自动迁移为加盐哈希。

### 4.2 笔记与待办管理

笔记编辑页支持标题、正文、图片、标签、AI 推荐标签、摘要和行动项。用户保存后，`NoteService` 将数据写入 RDB。首页通过 `NoteViewModel` 查询并转换为卡片数据展示。

待办支持三态管理：

- 待处理
- 已完成
- 已延期

`NoteCard` 中提供 Checkbox 和状态胶囊，`TodoDetailPage` 中提供更完整的待办编辑能力。状态变更后通过 `RdbStoreManager` 同步到数据库，列表与详情页保持一致。

### 4.3 搜索与筛选

搜索页支持关键词搜索、日期范围筛选和标签筛选。搜索结果复用 `NoteCard` 展示，待办状态可以在搜索结果中直接操作。宽屏下搜索页限制最大宽度并居中，搜索结果使用响应式网格展示。

### 4.4 图片功能

`ImageService` 封装 PhotoViewPicker 和图片权限申请。编辑页可选择或清除图片，`NoteCard` 根据断点调整图片高度，使手机和平板显示比例更合理。

### 4.5 本地 AI 整理功能

本地 AI 功能包括：

- 关键词提取
- 推荐标签
- 摘要生成
- 行动项提取
- 时间识别
- 优先级推断

其中关键词提取和句子评分由 C++ Native 模块实现，推荐标签、摘要压缩和行动项规则由 ArkTS 实现。近期迭代中修复了多个边界问题，例如：

- `明天记得买咖啡` 不再生成 `天记` 标签。
- 生活购物场景中不再出现 `备去`、`要买` 等跨词 n-gram 噪声。
- 多个行动项会生成多个待办预览卡片。
- 纯记录类文本不再强制生成待办。

### 4.6 云端 AI 整理功能

云端 AI 通过 `HttpUtil.postJson()` 调用通义千问兼容接口，请求格式类似 OpenAI Chat Completions。`NoteService` 负责构造结构化 Prompt，并要求云端返回 JSON：

```json
{
  "summary": "...",
  "keywords": ["..."],
  "actionItems": ["..."],
  "suggestedTags": ["..."]
}
```

解析逻辑支持直接 JSON、`data` 嵌套格式以及 `choices[0].message.content` 格式。如果云端请求失败，页面显示错误提示，本地 AI 能力仍可继续使用。

## 五、跨语言 Native 实现

### 5.1 C++ 构建配置

模块级 `entry/build-profile.json5` 添加了 Native 构建配置：

```json5
"externalNativeOptions": {
  "path": "./src/main/cpp/CMakeLists.txt",
  "arguments": "",
  "cppFlags": "",
  "abiFilters": [
    "arm64-v8a",
    "x86_64"
  ]
}
```

`CMakeLists.txt` 构建共享库 `text_analyzer`：

```cmake
add_library(text_analyzer SHARED
    text_analyzer.cpp
    crypto_utils.cpp
    napi_init.cpp
)

target_link_libraries(text_analyzer PUBLIC
    libace_napi.z.so
    libhilog_ndk.z.so
)
```

同时配置 `arm64-v8a` 和 `x86_64`，解决真机和模拟器 ABI 不一致导致的安装失败问题。

### 5.2 C++ 文本分析模块

`text_analyzer.cpp` 完成以下工作：

1. UTF-8 解码，确保中文字符边界正确。
2. 对中文文本进行 2-gram 和 3-gram 切分。
3. 过滤标点、纯数字、长度过短、停用词和跨词边界噪声。
4. 计算 TF-IDF 风格权重，结合预设 IDF 提升领域词权重。
5. 按权重排序返回 top K 关键词。
6. 将文本按句号、问号、感叹号、分号等切分句子。
7. 根据句长、关键词命中、摘要暗示词和长度惩罚对句子评分。
8. 返回得分最高的句子，供 ArkTS 层进一步压缩为摘要。

### 5.3 C++ 密码加密模块

`crypto_utils.cpp` 实现：

- SHA-256 哈希
- 16 字节随机盐值生成
- 十六进制编码
- 密码验证
- 常量时间字符串比较

密码存储格式为：

```text
64位hex哈希:32位hex盐值
```

该实现避免明文密码落盘，并兼容旧版本明文账号迁移。

### 5.4 NAPI 桥接

`napi_init.cpp` 注册了以下 Native 方法：

| 方法 | ArkTS 调用 | 功能 |
| --- | --- | --- |
| `extractKeywords` | `NativeTextAnalyzer.extractKeywords()` | 提取关键词 |
| `scoreSentences` | `NativeTextAnalyzer.scoreSentences()` | 评分摘要候选句 |
| `hashPassword` | `NativeCrypto.hashPassword()` | 计算密码哈希 |
| `generateSalt` | `NativeCrypto.generateSalt()` | 生成随机盐值 |
| `verifyPassword` | `NativeCrypto.verifyPassword()` | 验证密码 |

ArkTS 通过如下方式导入动态库：

```typescript
import textAnalyzer from 'libtext_analyzer.so'
```

然后在 `NativeTextAnalyzer.ets` 和 `NativeCrypto.ets` 中封装异常处理，确保 Native 调用失败时不会导致页面崩溃。

### 5.5 跨语言实现效果

跨语言实现带来的效果包括：

1. 中文关键词不再依赖空格切分，能够在无空格中文文本中提取有效关键词。
2. 文本分析逻辑从 ArkTS 迁移到 C++ 后，便于后续优化算法和复用。
3. 密码加密由 C++ 实现，减少 ArkTS 侧明文处理时间，提高安全性。
4. 通过 NAPI 将 Native 能力封装成 ArkTS 工具类，业务层调用方式保持简洁。

## 六、跨设备实现与效果

### 6.1 响应式断点系统

项目新增 `Breakpoint` 枚举和 `ResponsiveConstants` 常量，断点分为：

| 断点 | 宽度范围 | 布局策略 |
| --- | --- | --- |
| SM | 0-599vp | 手机单列 |
| MD | 600-839vp | 小平板/横屏，两列或居中表单 |
| LG | 840vp 及以上 | 平板三列或列表-详情双栏 |

`BreakpointUtil` 使用 `AppStorage` 保存全局断点状态，并提供页面 padding、网格列数、图片高度、文本预览行数等辅助方法。`EntryAbility` 在窗口创建和尺寸变化时更新断点，页面通过 `@StorageProp('currentBreakpoint')` 获取当前断点。

### 6.2 手机端效果

手机端保持单栏体验：

- 首页使用 Tab 区分“我的笔记”和“待办事项”。
- 笔记卡片单列展示。
- 点击卡片通过路由进入独立编辑页。
- 新建笔记和新建待办通过独立页面完成。
- 页面宽度充分利用屏幕，避免多栏导致内容拥挤。

### 6.3 平板端效果

LG 断点下首页使用列表-详情结构：

- 左侧栏宽度约 360vp，展示筛选区、笔记列表和待办列表。
- 右侧区域嵌入 `NoteEditPage` 或 `TodoDetailPage`。
- 点击左侧卡片不会跳转页面，而是直接切换右侧详情。
- 编辑标题、正文、图片和标签时，左侧卡片可以实时更新预览。
- 保存或删除后刷新列表并保持合理的选中状态。

### 6.4 跨设备关键实现

项目在跨设备实现中解决了以下问题：

1. 不能简单等比拉伸：通过断点控制列数、padding、表单最大宽度和图片高度。
2. 手机与平板导航方式不同：手机使用路由跳转，平板使用嵌入编辑。
3. 平板左侧列表需要实时预览草稿：通过 `onDraftChange` 回调更新父组件状态。
4. ArkUI `ForEach` 复用导致卡片不刷新：将标题、正文、图片和标签纳入 key。
5. ArkUI V1 `@Prop` 不适合函数回调：回调改为普通组件属性。

## 七、开发过程中遇到的问题及解决方法

### 7.1 中文关键词切分效果差

问题：最初本地 AI 使用空格切分文本。中文文本通常没有空格，导致整段内容被当成一个词，关键词几乎不可用。

解决：使用 C++ 实现 UTF-8 中文 n-gram 分词，并加入停用词过滤、领域词权重和 TF-IDF 风格排序。ArkTS 层通过 NAPI 调用 Native 模块。

### 7.2 Native 编译中 `LOG_DOMAIN` 和 `LOG_TAG` 冲突

问题：`hilog/log.h` 中已经定义 `LOG_DOMAIN` 和 `LOG_TAG` 宏，C++ 文件中使用同名常量后被宏替换，导致编译失败。

解决：将自定义日志常量改名为 `TEXT_ANALYZER_LOG_DOMAIN` 和 `TEXT_ANALYZER_LOG_TAG`，避免宏冲突。

### 7.3 安装时报 ABI 不匹配

问题：Native 动态库最初只生成 `arm64-v8a`，在 x86_64 模拟器上安装时报 “Abi type supported by the device does not match”。

解决：在 `entry/build-profile.json5` 中增加：

```json5
"abiFilters": ["arm64-v8a", "x86_64"]
```

并清理 `.cxx` 缓存后重新构建。

### 7.4 平板点击卡片闪退

问题：平板分栏中给子组件传递 `onSaved`、`onDeleted`、`onDraftChange` 回调时，使用了 `@Prop`，ArkUI V1 不支持函数类型 `@Prop`，导致运行时崩溃。

解决：将函数回调改为普通组件属性，避免状态装饰器处理函数对象。

### 7.5 平板左侧卡片不实时刷新

问题：编辑右侧详情时，左侧卡片没有立即更新标题和正文。

原因：`ForEach` key 只包含 ID 和状态，ArkUI 复用了旧节点。

解决：将标题、正文、图片、标签等预览字段纳入 key，草稿变化后强制刷新对应卡片。

### 7.6 多待办只显示一个预览

问题：AI 提取出多个行动项后，预览区域只显示一个待办卡片，用户以为只会保存一条。

解决：`NoteEditPage` 的预览区域改为复用 `buildGeneratedTodoInputs()`，通过 `ForEach` 展示多张待办卡片；按钮文案改为“保存 N 条待办事项”。

### 7.7 纯记录文本被强制生成待办

问题：读书笔记、观点摘录等纯记录文本没有明确行动项，但旧逻辑会把原文作为待办。

解决：删除行动项抽取的整句 fallback。页面层也不再使用摘要或正文兜底创建待办。若行动项为空，按钮显示“无可保存待办”，并提示“未识别到明确待办事项”。

### 7.8 短文本产生奇怪标签

问题：输入“明天记得买咖啡”时，n-gram 切出 `天记` 这类跨词边界片段，并进入推荐标签。

解决：在 C++ 停用词中加入 `天记`、`明天记`、`记得买` 等跨边界噪声；ArkTS 标签层也加入兜底停用词，双层过滤。

## 八、HapTest 测试结果与验证

### 8.1 测试目标

HapTest 测试围绕前四部分集成后的主链路展开，重点验证：

1. Native 文本分析和 Native 密码加密能与 ArkTS 正常协作。
2. 登录注册、笔记编辑、待办状态、搜索筛选等核心功能可运行。
3. 手机和平板布局能够按断点正确切换。
4. 平板列表-详情分栏无崩溃，保存和删除后列表状态正确。

### 8.2 测试策略一：核心功能集成冒烟

| 用例编号 | 用例名称 | 操作步骤 | 期望结果 | 当前结果 |

### 10.4 完整对话记录

以下为本次实验报告整理阶段围绕项目架构、Native 能力、响应式适配、集成测试与后续优化形成的 AI 使用问答整理记录。

**问题 1：请你概括这个 HarmonyOS 待办/笔记项目的整体架构。**

AI 回答：这个项目是一个基于 HarmonyOS Stage 模型的待办/笔记应用，整体可以分成五层。第一层是页面层，主要在 `entry/src/main/ets/pages` 下，包括登录、注册、首页、笔记编辑、待办详情和搜索结果页。第二层是组件层，在 `components` 下封装 `NoteCard`、`TagChip`、`TodoStatusSwitch`、`CommonEmptyView` 等通用 UI。第三层是业务服务层，`NoteService` 负责笔记 CRUD、标签解析和 AI 整理，`ImageService` 负责图片选择，`UserViewModel` 负责注册登录和密码校验。第四层是数据层，`RdbStoreManager` 使用 RDB 存储 notes 和 todos 两张表。第五层是 Native 能力层，通过 C++ NAPI 提供关键词提取、句子评分、密码哈希、盐值生成和密码验证能力。整体上，它是 ArkTS 页面、RDB 本地数据、C++ Native 能力和响应式布局结合的一体化应用。

**问题 2：Native C++ 模块在这个项目里起到了什么作用？**

AI 回答：Native C++ 模块主要承担两个高频、偏底层的能力：文本分析和密码安全。文本分析部分通过 `text_analyzer.cpp` 实现 UTF-8 中文处理、n-gram 分词、关键词排序和句子评分，再通过 NAPI 暴露给 ArkTS 的 `NativeTextAnalyzer`。密码安全部分通过 `crypto_utils.cpp` 实现 SHA-256 哈希、随机盐值生成和密码验证，再通过 `NativeCrypto` 给 `UserViewModel` 使用。这样做的好处是将相对复杂、性能要求更高的逻辑下沉到 C++，ArkTS 侧只保留清晰的业务调用入口。

**问题 3：为什么密码加密要使用盐值和哈希，而不是直接保存明文？**

AI 回答：直接保存明文密码风险很高，一旦本地偏好数据被读取，用户密码就会被直接泄露。使用哈希后，系统保存的是不可逆摘要，登录时只需要把用户输入的密码和盐值再次计算哈希，再与保存的哈希比对。盐值的作用是让相同密码在不同用户或不同注册记录中得到不同哈希，降低彩虹表和批量撞库攻击的风险。这个项目中注册时保存 `hash:salt`，登录时通过 NativeCrypto 校验，并保留旧明文密码自动迁移逻辑，兼顾安全性和兼容性。

**问题 4：响应式断点系统是怎么支持手机和平板适配的？**

AI 回答：项目在 `Constants.ets` 中定义了 SM、MD、LG 三类断点，并在 `BreakpointUtil` 中根据窗口宽度计算当前断点。`EntryAbility` 在窗口创建和尺寸变化时更新断点状态，页面通过 `@StorageProp('currentBreakpoint')` 获取当前断点。不同断点会影响页面 padding、Grid 列数、图片高度、正文预览行数和表单最大宽度。手机端以单列和路由跳转为主，平板端则使用多列网格和列表-详情双栏，从而避免简单拉伸导致的大屏空间浪费。

**问题 5：平板分栏架构相比普通页面跳转有什么优势？**

AI 回答：普通页面跳转更适合手机小屏，因为一次只展示一个主要任务。但在平板上，如果仍然使用单页跳转，大屏空间会被浪费，用户也需要频繁返回列表。分栏架构把左侧列表和右侧详情同时展示，用户点击左侧卡片后，右侧直接切换到笔记编辑或待办详情，列表仍然可见。这样可以提升浏览、比较和连续编辑的效率，也更符合平板横屏使用习惯。项目中还通过嵌入模式和草稿回调实现了右侧编辑、左侧预览同步。


**问题 6：`RdbStoreManager` 中 notes 表和 todos 表为什么要分开设计？**

AI 回答：notes 表保存笔记和待办都需要的通用信息，例如标题、正文、图片、标签、摘要、行动项和创建时间；todos 表保存待办特有的信息，例如待办状态、截止时间和优先级。分开设计可以避免 notes 表无限膨胀，也方便后续扩展提醒、重复任务、截止日期等待办专属能力。同时，普通笔记和待办仍然可以通过 `noteId` 关联，共用卡片展示、搜索和编辑逻辑，在复用和扩展之间取得平衡。

**问题 7：本地 AI 摘要和关键词提取的基本流程是什么？**

AI 回答：本地 AI 流程从用户在编辑页输入标题和正文开始。点击本地 AI 整理后，页面调用 `NoteService.generateLocalAiSuggestion(title, content)`。服务层先拼接标题和正文，再调用 `NativeTextAnalyzer.extractKeywords` 获取关键词，并调用 `NativeTextAnalyzer.scoreSentences` 获取摘要候选句。随后 ArkTS 规则层继续处理标签推荐、摘要压缩、行动项抽取、时间信息识别和优先级推断。最后页面展示关键词、推荐标签、摘要和待办行动项，用户可以选择保存为普通笔记或待办。

**问题 8：第五部分“集成测试与优化”主要应该验证哪些内容？**

AI 回答：第五部分重点不是新增单点功能，而是验证前四部分是否能组合成完整可运行应用。需要验证 Native 文本分析和密码加密是否能正确打包并被 ArkTS 调用，RDB 数据读写和待办状态同步是否稳定，响应式断点在手机和平板宽度下是否正确切换，平板分栏中的列表、详情和草稿预览是否同步。同时还需要整理 HapTest 测试策略、记录构建结果、修复明显集成问题，并把高频 UI 硬编码尺寸统一到常量中，保证后续维护更容易。

**问题 9：UI 硬编码尺寸改成 `UiConstants` 有什么好处？**

AI 回答：把硬编码尺寸改成 `UiConstants` 的好处是统一、可维护和可扩展。原来页面里到处出现 `28`、`56`、`44`、`16`、`14` 等数值，后续如果要调整标题字号、输入框高度或卡片内边距，需要在多个页面逐个搜索替换，很容易漏改。统一到 `UiConstants.TITLE_SIZE`、`CONTROL_HEIGHT`、`COMPACT_CONTROL_HEIGHT`、`CARD_PADDING`、`FIELD_RADIUS` 等常量后，语义更清楚，页面风格更一致，也方便响应式适配和设计规范迭代。

**问题 10：当前项目还有哪些可以继续优化的风险点？**

AI 回答：当前项目已经可以完成主要功能和 HAP 构建，但仍有一些后续优化点。第一，云端 AI API Key 仍然硬编码在常量文件中，应该迁移到安全配置或服务端代理。第二，RDB 和 Preferences 的部分调用存在未显式捕获异常的构建警告，可以统一增加错误处理。第三，`router`、`showToast`、`PhotoViewPicker` 等 API 存在废弃提示，后续应迁移到新版接口。第四，Native `.so` 导入缺少 `.d.ts` 声明文件，未来 SDK 校验更严格时可能受影响。第五，`NoteService` 体量较大，后续可以拆分 AI、标签、摘要和 CRUD 逻辑，降低维护成本。

#### 分工 3：响应式布局断点系统 AI 辅助对话记录

以下对话记录用于说明分工 3“响应式布局断点系统”开发过程中，开发者如何使用 AI 辅助完成需求拆解、方案设计、代码实现和验证思路整理。

**问题 1**

我：我的分工是“响应式布局断点系统”，这个任务在 HarmonyOS/ArkTS 中通常应该拆成哪些子任务？

AI：建议拆成六个部分：第一，在常量文件中定义断点枚举和响应式常量；第二，封装断点工具类，统一判断当前屏幕属于手机、小平板还是大平板；第三，在 Ability 或页面入口监听窗口宽度变化；第四，将首页列表从单列 `List` 改为可响应的 `GridRow/GridCol`；第五，让卡片组件根据断点调整图片高度和正文行数；第六，让编辑页、详情页和搜索页在平板上限制最大宽度并居中显示。这样拆分后，每个模块职责清晰，后续第四位同学做平板分栏时也可以复用断点系统。

**问题 2**

我：手机和平板适配时，断点应该如何设置？`600vp`、`840vp` 这样的阈值是否合理？

AI：合理。可以把 `0-599vp` 定义为 `SM`，用于手机单列布局；把 `600-839vp` 定义为 `MD`，用于横屏手机或小平板，两列布局；把 `840vp` 及以上定义为 `LG`，用于平板三列或分栏布局。`vp` 比像素更适合做布局判断，因为它更接近 ArkUI 实际布局单位，不会直接受设备 DPI 影响。

**问题 3**

我：ArkTS 中如何保存全局断点状态，让多个页面和组件都能读取当前是手机还是平板？

AI：可以使用 `AppStorage` 保存全局断点，例如保存键名 `currentBreakpoint`。页面和组件通过 `@StorageProp('currentBreakpoint')` 订阅该值。这样 `Index.ets`、`NoteCard.ets`、`NoteEditPage.ets` 等组件都能共享同一个断点状态，只要窗口尺寸变化后更新 `AppStorage`，所有订阅组件都会自动刷新布局。

**问题 4**

我：`AppStorage` 和 `@StorageProp` 在这个场景中怎么使用？

AI：启动时先执行 `AppStorage.SetOrCreate('currentBreakpoint', Breakpoint.SM)` 初始化默认断点。页面中声明 `@StorageProp('currentBreakpoint') currentBreakpoint: Breakpoint = Breakpoint.SM`。当窗口宽度变化时调用 `AppStorage.SetOrCreate('currentBreakpoint', nextBreakpoint)` 更新状态，页面中的 `currentBreakpoint` 会同步变化，然后通过辅助函数计算 padding、列数、图片高度等布局值。

**问题 5**

我：如何通过窗口宽度变化或页面区域变化更新当前断点？

AI：可以采用两层方案：第一层在 `EntryAbility.ets` 的 `onWindowStageCreate` 中获取主窗口尺寸，并监听 `windowSizeChange`；第二层在页面根组件上使用 `onAreaChange` 获取实际布局区域宽度。前者用于应用启动和窗口变化的兜底，后者更贴近 ArkUI 页面实际可用宽度。最终都调用 `BreakpointUtil.updateByWidth(width)` 统一更新断点。

**问题 6**

我：`EntryAbility.ets` 中适合放哪些断点初始化逻辑？

AI：`EntryAbility.ets` 适合做全局初始化。可以在 `onCreate` 中初始化断点状态，在 `onWindowStageCreate` 中读取主窗口宽度，并注册 `windowSizeChange` 监听。不要在 Ability 中写具体页面布局逻辑，Ability 只负责把屏幕变化同步到全局状态，具体怎么布局交给页面和组件根据断点自行决定。

**问题 7**

我：首页原来用 `List` 展示笔记卡片，如何改成手机单列、平板多列的 `GridRow/GridCol`？

AI：可以把原来的 `List` 外层换成 `Scroll`，内部使用 `GridRow`。`GridRow` 的总列数可以按断点动态设置，例如手机为 4，`MD` 为 8，`LG` 为 12；每个 `GridCol` 的 `span` 分别为 4、4、4 或统一通过工具函数换算为 12、6、4。这样手机一行一个卡片，`MD` 一行两个卡片，`LG` 一行三个卡片。卡片点击事件和状态回调保持原逻辑不变。

**问题 8**

我：`GridRow` 的 `columns` 和 `GridCol` 的 `span` 应该如何设计，才能实现 1 列、2 列、3 列切换？

AI：推荐用 12 栅格思路：`GridRow({ columns: 12 })` 时，手机每个卡片 `span=12`，两列时 `span=6`，三列时 `span=4`。如果为了和断点列数对应，也可以让工具函数返回 `columns * 4`，并让 `span` 在 `SM/MD/LG` 下分别为 `12/6/4`。核心原则是：`GridRow columns / GridCol span = 每行卡片数量`。

**问题 9**

我：`NoteCard` 如何根据断点调整图片高度？

AI：在 `NoteCard` 中订阅 `currentBreakpoint`，然后封装 `getImageHeight()`。例如 `SM` 返回 `160vp`，`MD` 返回 `180vp`，`LG` 返回 `200vp`。图片组件继续设置 `.width('100%')` 和 `.objectFit(ImageFit.Cover)`，只动态改变高度。这样手机端卡片紧凑，平板端图片展示更充分。

**问题 10**

我：`NoteCard` 如何根据断点调整正文预览行数？

AI：同样在 `NoteCard` 中通过断点计算 `maxLines`。例如 `SM` 显示 3 行，`MD` 显示 4 行，`LG` 显示 5 行。正文仍然保留 `textOverflow({ overflow: TextOverflow.Ellipsis })`，保证内容过长时不会撑开卡片。这样宽屏空间更大时能展示更多内容，手机端仍保持列表紧凑。

**问题 11**

我：编辑页 `NoteEditPage` 在平板上为什么不应该无限拉宽？最大宽度设置多少比较合适？

AI：表单页面如果在平板上铺满整屏，输入框会过长，阅读和编辑效率反而下降。因此平板端应该限制内容最大宽度并居中。笔记编辑页和待办详情页可以使用 `760vp` 左右作为最大宽度，既能容纳完整表单，又不会显得过宽。手机端则保持 `100%` 宽度，充分利用屏幕空间。

**问题 12**

我：`TodoDetailPage` 和 `SearchResultPage` 应该如何复用同一套响应式布局逻辑？

AI：把断点判断、页面 padding、网格列数、图片高度等公共逻辑放到 `BreakpointUtil` 和 `ResponsiveConstants` 中。`TodoDetailPage` 只需要复用表单最大宽度和页面 padding；`SearchResultPage` 复用页面 padding、最大宽度、`GridRow/GridCol` 列数和 span。这样不同页面不需要重复写断点判断代码。

**问题 13**

我：搜索结果页如何从单列 `Column` 改成响应式网格？

AI：保留搜索筛选区不变，只把结果区从 `Column({ space: 12 })` 改成 `GridRow({ columns, gutter })`。每条搜索结果包在 `GridCol({ span })` 中，内部仍然使用 `NoteCard`。这样手机端仍显示单列，平板端搜索结果自动变成两列或三列，同时待办状态切换回调仍然可以沿用原来的 `handleToggleStatus` 和 `handleStatusChange`。

**问题 14**

我：`module.json5` 中是否需要增加 `tablet` 设备类型？为什么？

AI：需要。既然任务目标是手机和平板适配，`deviceTypes` 中只写 `phone` 会导致工程配置和多端适配目标不一致。建议将其改为 `["phone", "tablet"]`。这样从工程声明层面也说明应用支持平板设备，和响应式断点、平板分栏等代码实现保持一致。

**问题 15**

我：这次响应式改造涉及哪些文件？每个文件分别承担什么作用？

AI：主要涉及 `Constants.ets`、`BreakpointUtil.ets`、`EntryAbility.ets`、`Index.ets`、`NoteCard.ets`、`NoteEditPage.ets`、`TodoDetailPage.ets`、`SearchResultPage.ets` 和 `module.json5`。其中 `Constants.ets` 定义断点和常量，`BreakpointUtil.ets` 管理断点判断，`EntryAbility.ets` 初始化和监听窗口，`Index.ets` 实现首页响应式网格，`NoteCard.ets` 调整卡片尺寸，三个页面负责宽屏居中，`module.json5` 声明支持平板。

**问题 16**

我：如何在 DevEco Studio Previewer 中验证手机端单列布局？

AI：打开 `Index.ets` 或运行应用后，在 Previewer 中选择手机设备，例如宽度约 `360-430vp` 的配置。登录进入首页后观察“我的笔记”和“待办事项”两个 Tab，卡片应为单列展示。点击卡片仍应通过路由进入独立编辑页或待办详情页，说明手机端导航逻辑没有被平板布局改动影响。

**问题 17**

我：如何在 DevEco Studio Previewer 中验证平板端 2 列和 3 列布局？

AI：在 Previewer 中自定义设备宽度。若宽度约大于 `600vp` 且小于 `840vp`，首页和搜索结果应显示两列；若宽度大于等于 `840vp`，应显示三列。可以用 `Resolution * 160 / DPI` 粗略估算 `vp` 宽度，例如 `2560px / 480dpi * 160 ≈ 853vp`，会进入 `LG` 断点，预期显示三列。

**问题 18**

我：如何根据分辨率和 DPI 估算设备的 vp 宽度？

AI：可以用公式 `vp宽度 = 像素宽度 * 160 / DPI`。例如当前手机配置宽度 `1272px`、`DPI=540`，则 `1272 * 160 / 540 ≈ 377vp`，属于手机 `SM` 断点。若想测试大平板，可设置 `2560 x 1600`、`DPI=480`，宽度约 `853vp`，会进入 `LG` 断点。

**问题 19**

我：如果 DevEco Studio 编译报 `@StorageProp` 或 `GridRow` 相关错误，应该优先检查哪里？

AI：先检查 `@StorageProp` 的键名是否为字符串字面量，例如 `@StorageProp('currentBreakpoint')`，再检查 `AppStorage` 是否在启动时初始化。`GridRow/GridCol` 报错时，重点检查 `columns`、`span` 参数是否为数字，`ForEach` 闭合结构是否完整，原来的 `ListItem` 是否已经移除。还要确认新增的 `BreakpointUtil.ets` 路径和 import 是否正确。

**问题 20**

我：实验报告中应该如何描述“AI 在本任务中承担的工作”？

AI：可以描述为：AI 主要承担开发辅助角色，用于帮助拆解响应式适配任务、解释 HarmonyOS 断点设计、给出 `AppStorage` 与 `@StorageProp` 的状态共享方案、建议 `GridRow/GridCol` 多列布局实现方式、整理 DevEco Previewer 验证步骤。最终代码由开发者结合项目结构手动修改、调试和验证，AI 不参与应用运行时功能。

### 10.5 开发辅助 AI 与应用内 AI 功能的区别

开发辅助 AI 是开发阶段使用的豆包，用于帮助开发者理解任务、设计方案、解释算法和准备测试用例。它不会被打包进应用，也不会在用户使用应用时直接运行。

应用内 AI 功能是本项目实际实现并提供给用户的功能，包括本地 AI 自动提取、云端 AI 整理、关键词推荐、摘要生成和待办事项提取等。应用内 AI 功能运行在 HarmonyOS 应用中，通过 ArkTS、Native C++ 模块和相关业务逻辑共同完成。
## 十一、总结

本项目完成了一个面向手机和平板的 HarmonyOS 灵感笔记应用。基础功能上，应用支持用户注册登录、笔记管理、待办管理、搜索筛选、图片选择和 AI 整理。Native 增强方面，项目将中文关键词提取、句子评分和密码加密迁移到 C++，通过 NAPI 被 ArkTS 调用，体现了跨语言协同开发能力。跨设备适配方面，项目通过断点系统和 SideBarContainer 实现手机单栏、平板多栏和平板列表-详情双栏，避免了简单拉伸。

开发过程中解决了中文分词、Native ABI、NAPI 宏冲突、ArkUI 回调属性、平板列表刷新、多待办预览、无待办场景误生成等问题。测试方面，项目完成了本机 HAP 构建验证、Native 密码加密验证、手机和平板模拟器验证，并整理了 HapTest 主链路测试方案。后续可继续完善自动化真机测试记录、Native `.d.ts` 类型声明、云 AI 密钥安全管理和本地词典外置能力。
